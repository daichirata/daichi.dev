# daichi.dev 執筆スタイル

このリポジトリで記事を起こす・直す・体裁を整える時は、必ずこの SKILL を読み込んでから手を動かす。著者には言語化されていない癖がいくつかあり、そこを外すと一目で「LLM が書いた」と分かる文章になる。

## 起動前に必ずやる事

1. `_posts/` 配下の最新の技術記事を 1〜2 本流し読みする。今のキャリブレーション対象は次の三つ:
   - `_posts/2026-05-22-gcsproxy-range.md` ← 最新かつ著者の編集を最も新鮮に反映している、第一参考
   - `_posts/2023-01-19-golang-x-time-rate.md`
   - `_posts/2017-03-21-circuit-breaker.md`
2. 直前の記事の語り口・テンポ・体言止めの頻度を、頭の中で 1 段落くらい真似してみる。
3. その上で初稿を書き始める。

## ボイス

- **一人称は「自分」のみ。**「僕」「私」は使わない。
- 反応・感想は淡めに、断定的に。
  - 使って良い: 「個人的に面白いと思う」「〜という印象」「〜という感じ」「〜と思う」
  - **使わない**: 「なるほど」「それなら」「ということで〜と思いました！」のような相槌・接続フィラー。書きそうになったら、その語を消して内容で語り出す形に書き直す。
- ヘッジは控えめ。`〜と思う` で十分なところに `〜なのではないかと考えられる` などを足さない。

## 文の繋ぎ方

- **連用形で文を繋いで一文にまとめる癖がある**。LLM が短い 2 文に分けがちな所を、`〜て、` `〜ので、` `〜のだけど、` `〜があり、` で繋いで 1 文にする方が著者の文体に近い。
- **括弧書きの補足はなるべく本文に溶かす**。`(後述する)` のような独立アサイドより、`gcsproxy では後述するように後者寄せにした` のように本文に組み込む方を選ぶ。`( )` は **定義的な短い言い換え** だけ使う (例: ``If-Range`` (リクエストヘッダのバージョンが一致したら range、違ったら full body を返す機能))。
- **登場人物に台詞を喋らせない**。「サーバーが『そんなの知らないよ』と返しても」のような演出は避け、「サーバーが無視して 200 でフルボディを返しても」と直接書く。

## 内容の節度

- **外部システムの「意図」を推測しない**。観測した事実 (例: 「GCS は gzip オブジェクトに Range を投げると解凍済み全体を返してきた」) を述べた所で止める。「〜という挙動になっているのだと思う」のような勝手な動機付けは付け足さない。事実だけが事実。
- **データがある所は出す**。実機検証のログ、プローブの出力、`go test` の結果。手元に出力が無い主張は、断定を弱めるか、書かない。

## 記事の構造

- frontmatter は `title:` だけ。`layout:` `date:` などは Jekyll が自動で処理する。
- 冒頭はプロジェクト紹介 + 何故書くか。「インターネットには〜」「近年〜」のような一般論で始まる導入は書かない。
- 見出しは `##` 主体。`###` は本当に階層が必要な時だけ。
- コードブロックは言語タグ付き fenced block (` ```go ` ` ```bash ` 等)。識別子・ヘッダ名・コマンド名は inline `code`。
- 挙動の場合分け表や API のオプション比較は markdown テーブルでまとめて良い。最新記事でも使っている。
- 締めは短い `## おわり` で、残した課題やトレードオフを正直に書く。「いかがでしたか？」「読んでくれてありがとう」のような型は使わない。

## 公開ワークフロー

1. 下書きは `_drafts/<YYYY-MM-DD>-<slug>.md` に置く。
2. 公開時は `_posts/` に同名で移動する (`git mv` または `mv` + `git add`)。frontmatter のファイル名の日付と公開日を一致させる。
3. コミットメッセージは **わざと短い** (`Add post` 等)。`git log --oneline` を見て今の慣例に合わせる。
4. **このリポジトリでは Co-Authored-By trailer は付けない**。
5. 記事のステージング・コミット時に **無関係な unstaged ファイル** (`.github/workflows/`, `Gemfile.lock`, `package.json`, `yarn.lock` 等) を巻き込まない。`git add` は記事ファイルを名指しで行う。
6. `git push` は明示的に頼まれない限り自動で行わない。

## チェックリスト (執筆後・コミット前)

- [ ] 一人称は「自分」だけになっているか
- [ ] 「なるほど」「それなら」など相槌系のフィラーが残っていないか
- [ ] 短文連発になっていないか (連用形で繋げる余地はないか)
- [ ] `( )` の独立補足は本文に溶かせないか
- [ ] 観測した事実の後に勝手な意図推測を付け足していないか
- [ ] 締めが `## おわり` で、定型挨拶になっていないか
- [ ] `git status` で記事以外のファイルがステージに入っていないか
